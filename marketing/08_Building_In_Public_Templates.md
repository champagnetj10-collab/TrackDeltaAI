# "Building in Public" Post Templates

Reusable fill-in-the-blank templates for documenting real development progress. The whole point of building in public is specificity and honesty — these templates exist to make posting *fast*, not to make the content generic. Always replace bracketed sections with something true and specific; never publish a template with the brackets still vaguely paraphrased.

Read [01_Brand_Voice_and_Style_Guide.md](01_Brand_Voice_and_Style_Guide.md) first — building-in-public posts use the founder's own voice, slightly more candid than the standard brand-account tone.

---

## Template 1 — Weekly Ship Log (recurring, every week)

**Platforms:** X (thread or single post), LinkedIn (single post), short-form video version = SF-27 format

```
Shipped this week on TrackDelta AI:

✅ [specific thing #1 — one line, plain language, no jargon]
✅ [specific thing #2]
✅ [specific thing #3, or omit if only 2 things shipped — never pad the list]

Also hit a snag: [one honest thing that didn't go as planned, or "nothing broke this week" if genuinely true]

Next week: [the one thing you're most focused on]
```

## Template 2 — The Honest Metric Post

**Platforms:** X, LinkedIn

```
Real number, no spin: [metric] is at [number] as of today.

[One sentence of honest context — is this good, bad, expected, surprising?]

[What this number is telling you to do next, if anything]
```

*Use for: sign-ups, session-upload counts, retention rate, conversion rate, North Star metric progress. Never post a vanity metric (like raw follower count) dressed up as a business metric.*

---

## Template 3 — The Bug/Mistake Post

**Platforms:** X, LinkedIn, short-form video (SF-22 format)

```
Found a bug this week: [what happened, in plain language, no need to explain code to a non-technical audience].

Here's what could have gone wrong: [the actual stakes — e.g. "a driver's Driver DNA could've been silently lost"].

Fixed before it affected anyone / here's who it affected and how it was resolved: [honest status].

Why I'm posting this instead of quietly fixing it: [reason — usually: trust is built by showing the process, not hiding it].
```

*This is one of the highest-trust content types available to a technical product — use it whenever something real happens, don't manufacture bugs to post about.*

---

## Template 4 — The "Why We Said No" Post

**Platforms:** X, LinkedIn, short-form video (SF-24 format)

```
A feature request that keeps coming up: [feature name/description].

We're not building it (yet). Here's why: [the real reasoning — usually a conflict with a core principle, a sequencing decision, or a "not proven yet" reason].

What we're doing instead: [the actual current priority].
```

---

## Template 5 — The User Feedback Reaction Post

**Platforms:** X, LinkedIn, short-form video (SF-25 format)

```
Real feedback from this week (shared with permission):

"[direct quote — praise, criticism, or a mix]"

[Honest reaction — agree, disagree, surprised, already working on it]

[What changes as a result, if anything]
```

*Always get explicit permission before quoting a user, even anonymized. Never fabricate or composite a "user quote" — this is exactly the kind of thing that destroys trust instantly if discovered, and this audience checks.*

---

## Template 6 — The Milestone Post

**Platforms:** All — this is the one template type worth cross-posting everywhere simultaneously

```
[Milestone]: [specific number/fact — first 100 users, first paying customer, first driver to report a measurable lap-time improvement, 90 days since launch, etc.]

[One paragraph: what it took to get here, honestly, including what was harder than expected.]

[One paragraph: what's next.]

Thank you to everyone who [specific thing — uploaded a session, gave feedback, shared it with a teammate].
```

## Template 7 — The Technical Deep-Dive Post (for a more technical audience segment, X/LinkedIn thread format)

```
How [specific system] actually works in TrackDelta AI, thread:

1/ [The problem this system solves]
2/ [The approach taken, in plain-enough language that a non-engineer sim racer can still follow the shape of it]
3/ [One honest trade-off or limitation of the current approach]
4/ [What it would take to improve it further, if relevant]
5/ [Tie back to why this matters for the driver, not just as a technical curiosity]
```

*Good candidates: how the .ibt parser works, how EWMA updates Driver DNA, why raw telemetry never touches the LLM, how confidence scoring is calculated.*

---

## Template 8 — The Roadmap Transparency Post

**Platforms:** X, LinkedIn — pair with LF-20 style video periodically

```
What's actually next for TrackDelta AI, no vague roadmap-speak:

Building next: [1-3 specific, real near-term items]
Deliberately not building yet: [1-2 specific deferred items, with the one-line reason]
Still deciding: [anything genuinely undecided — it's fine to say so]
```

## Posting Cadence Guidance

| Template | Frequency |
|---|---|
| Weekly Ship Log | Every week, same day (pick one, e.g. Friday) |
| Honest Metric Post | Every 2 weeks, or at natural milestones |
| Bug/Mistake Post | Whenever something real happens — don't force it |
| "Why We Said No" | Every 3-4 weeks |
| User Feedback Reaction | Weekly if there's real feedback to share, don't skip weeks by inventing filler |
| Milestone Post | Whenever a real milestone hits |
| Technical Deep-Dive | Every 3-4 weeks, alternating with education content so the account doesn't feel like a dev blog exclusively |
| Roadmap Transparency | Monthly |

## The One Rule That Matters More Than Any Template

If a post requires stretching the truth to fill in a template, don't post that week. A missed week of building-in-public content costs nothing. A single caught exaggeration costs the credibility the entire format is built on.
