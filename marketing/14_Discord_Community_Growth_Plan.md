# Discord Community Growth Plan

This is a plan for growing **TrackDelta AI's own Discord server**. It's distinct from [10_Creator_Community_Hashtag_List.md](10_Creator_Community_Hashtag_List.md) §3, which covers engaging with *other* existing Discord communities — do that first; it's where the initial members of TrackDelta's own server will come from.

---

## 1. Should This Exist Yet? (Honest Gate Before Building It)

Don't launch a dedicated Discord server on day one of the product's own launch — an empty server is worse than no server; it signals low activity to the first people who find it. Launch it once there's a real base of engaged users (a reasonable trigger: 50-100 real sign-ups, or clear signal from existing channels that people want a direct line to each other and to the team). Until then, funnel community energy into the existing sim racing Discords per file 10.

---

## 2. Server Structure

```
WELCOME
  #welcome-and-rules
  #announcements (read-only, team posts only)
  #introduce-yourself

DEBRIEFS & FEEDBACK
  #share-your-debrief        (drivers post real debrief screenshots/reactions)
  #feature-requests
  #bug-reports
  #feedback-general

COMMUNITY
  #general
  #setups-and-rigs
  #off-topic

SUPPORT
  #help                      (direct product support, monitored by the team)

TEAM
  #building-in-public        (founder posts real progress — mirrors file 08 templates, native to this audience)
```

Keep it small and focused at launch — a server with 20 empty channels feels dead even with 50 active members; a server with 6 channels and the same 50 members feels alive.

---

## 3. Roles & Moderation

| Role | Who | Purpose |
|---|---|---|
| Team | Founder + anyone helping run the product | Full permissions, visibly tagged |
| Pro Member | Auto-assigned (or manually, pre-automation) to Pro subscribers | Small status marker, access to a Pro-only channel if/when that makes sense |
| Verified Driver | Anyone who's connected their account / posted a real debrief | Signals genuine participants vs. drive-by joiners |
| Regular | Default | |

**Moderation stance:** Light-touch at small scale — the founder should be the visible, present moderator early on, not a hired mod team. This matches the "Coach Dave Academy"-style authenticity this audience responds to (see file 01's audience notes) — a founder who's genuinely present in their own community is a trust asset, not just an operational necessity.

---

## 4. Growth Tactics

1. **Seed with real users first.** Invite the first cohort of actual paying/active users personally, one at a time if needed, before any public "join our Discord" push — a server that opens with 15 genuinely engaged early users feels alive to member #16 in a way an empty server never does.
2. **Cross-post the invite** in every relevant bio/link-in-bio (file 07's bio template should get a Discord link added once the server exists) and at the bottom of the weekly ship-log posts (file 08 template 1).
3. **Give Discord-exclusive value**, not just a redundant feed of what's already public: earliest access to new features, the ability to submit a session directly to the founder for the "we analyzed a subscriber's session" content format (LF-08), and direct input into roadmap prioritization discussions.
4. **Weekly presence ritual:** the founder should post something in `#building-in-public` at least weekly (reuse file 08's templates directly) and spend a set block of time (e.g., 30 min, same day each week) actively replying in `#help` and `#feedback-general` — visible, scheduled presence beats sporadic large bursts.
5. **Highlight real members.** Feature a real driver's debrief or improvement story (with permission) periodically in `#announcements` — this is the Discord-native version of the "most improved DNA" content format (SF-88).
6. **Connect it to referrals.** If the referral program (file 15) ships, Discord is a natural place to surface leaderboards or milestones for it — but only if genuinely opt-in and non-spammy.

---

## 5. What NOT to Do

- Don't buy or fake member counts — a server showing "500 members, 3 online" reads as dead and damages trust worse than a small, honestly-active server
- Don't over-moderate early — heavy-handed rules in a tiny community feel bureaucratic and cold; add structure as the server actually needs it, not preemptively
- Don't let `#announcements` become the only active channel — if the team is the only one posting anywhere, it's a broadcast channel, not a community
- Don't neglect it once it's launched — an abandoned-feeling Discord (long team silence, unanswered questions) is worse for trust than not having one at all

---

## 6. Metrics to Watch (ties into file 21)

- Weekly active members (not just total member count)
- Response time to `#help` posts
- Number of real, unprompted debrief shares in `#share-your-debrief`
- Retention: are members from month 1 still active in month 3?

If weekly active members flatlines or declines for several weeks running with no clear external cause, that's the signal to invest founder time back into presence (§4.4) before investing in more growth tactics — an inactive server won't be fixed by more invites.
